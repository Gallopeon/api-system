API Transformation Rules: A Configurable Privacy-Preserving Middleware Architecture

Control Plane Project · Demonstration & Technical Reference
Abstract. We present a configurable API transformation middleware supporting field-level whitelisting, renaming, masking, computed literal injection, conditional rule evaluation, gray-release A/B variant routing, pagination templating, and request/response validation. This interactive demonstration showcases six pre-configured transformation rules across two domains — Healthcare (HIPAA compliance, drug inventory, telemedicine) and Social (user feeds, messaging, user profiles) — with a complete client-side pipeline simulator for real-time experimentation. No server is required; all data and logic are self-contained.

3. Transformation Pipeline Methodology

Each API response record passes through the stages below in strict sequence. The pipeline is configurable per API endpoint and supports gray-release canary deployments via deterministic hash-based bucket assignment.

#StageDescription
1Gray ReleaseDeterministic variant selection via stable_bucket(bucket_field) hash (mod 100) with weighted assignment. Force-variant override supported for testing.
2WhitelistField-level allowlist filtering — only enumerated fields pass through; all others discarded.
3Conditional RulesExpression-evaluated rules (== != > < >= <= && || contains() exists()) that inject, remove, mask, or rename fields when conditions match. stop_after_match short-circuits subsequent rules.
4RenameField key remapping for API versioning and schema evolution.
5MaskPII/sensitive field redaction — replaces values with a masking placeholder.
6Computed LiteralsStatic metadata injection: version strings, compliance markers, system identifiers.
7Remove NullsStrips null/undefined fields from output to reduce payload size.
